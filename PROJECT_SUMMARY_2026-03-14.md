# 2026-03-13 至 2026-03-14 工作总结

本文档总结从昨天到今天围绕 `QQ bot 桥接 + codexBot Web 控制台 + 仓库发布` 所做的功能改进、代码优化和项目整理工作。

## 一、桥接链路问题修复

### 1. 修复 Codex 首轮参数冲突

- 修复 `reasoning.effort = minimal` 与 `web_search` 不兼容导致的 400 报错
- 将首轮轻聊策略改成更稳妥的只读组合
- 避免 QQ 侧收到底层 `invalid_request_error`

### 2. 优化错误提示归一化

- 过滤 PowerShell / Codex 内部噪声
- 将底层额度限制、网络超时等错误转成更短的中文提示
- 避免 QQ 端收到一长段 `stderr`

### 3. 修复过程文本误回传

- 防止 Codex 失败时把“我先检查/我先看下”这类中间过程误当最终回复发给 QQ
- 改成统一错误提示或清晰结论

### 4. 修复执行类消息误判

- 收紧执行类请求识别逻辑
- 避免“问题现在修复了吗？”被误判成执行请求而卡住

### 5. 修复桥接运行时异常

- 修复 `previewState is not defined` 等运行错误
- 修复 `bridge-manager` 中被乱码污染导致的变量丢失问题
- 增强会话级异常兜底和日志可读性

### 6. 增强超时与重试

- 放宽部分请求的超时窗口
- 为瞬时网络抖动增加一次自动重试

## 二、QQ 交互体验优化

### 1. 回复风格优化

- 默认更简洁
- 优先直接下结论
- 减少“我先检查/我先看看”式铺垫
- 更适合手机阅读

### 2. 执行前先规划

- 对执行型请求先返回规划框架
- 用户确认后再执行
- 支持“确认 / 开始执行 / proceed”一类继续指令

### 3. 状态类问题本地直答

- 像“问题现在修复了吗？”这类状态问句，优先基于本地 bridge 状态回答
- 降低模型自由发挥导致的泛化回复

## 三、Web 控制台从无到有搭建

## 六、总体结果

到目前为止，这套系统已经从“QQ 桥接调不稳、报错不清晰、缺少监控界面”的状态，推进到：

- QQ bot 可正常桥接本地 Codex
- 错误提示更清晰
- 支持规划后执行
- 有完整的 Web 控制台用于设置、监控和排障
- 有可发布、可协作的独立仓库 `codexBot`

## 七、当前建议的日常使用方式

推荐的实际使用方式是：

- 在 QQ 上发消息与 bot 交互
- 在 Web 控制台里查看状态、日志、审计和对话详情
- 通过 doctor / repair / threshold / export 做日常运维与留档
